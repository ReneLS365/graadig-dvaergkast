# V15-03 Wave Director Design

Status: **design complete; implementation pending**. This document is docs-only and defines the required Survival-only implementation shape for `V15-03-impl — Wave director implementation`.

## Scope and non-goals

The V15-03 wave director replaces Survival's distance-derived wave presentation with an explicit deterministic wave state machine. It must not change Campaign, Daily Mine, Duel, Random, Practice, or Chaos.

In scope for the later implementation PR:

- Survival-only wave state/data.
- Explicit wave phases: `telegraph` → `spawn` → `peak` → `lull`.
- Deterministic scheduling from the run seed.
- HUD data for phase, current wave, and next-wave progress.
- Scheduling of existing hazards/enemies and existing crates only where already allowed by current Survival rewards.
- Mine Core pressure through timing, lane position, and breach-focused pressure from existing hazards.

Out of scope for V15-03 implementation:

- New enemy types, new rewards, bosses, elite variants, difficulty tiers, cards, workshop, labs, or draft upgrades.
- Economy, reward, currency, or balance-value changes outside documented wave cadence.
- Save schema/data changes.
- Campaign, Daily Mine, Duel, or leaderboard behavior changes.
- Generated `dist/` edits that are not produced by build scripts.

## Current behavior to transition from

Current Survival waves are derived from distance: `survivalWave = Math.floor(run.distance / 900) + 1`, and the V15-02 HUD shows next-wave progress from `run.distance % 900`. Hazard placement is pre-generated by `makeHazards(seed, modeMods)` across the track, then normal collision, kill, breach, and reward logic consumes those hazards as the camera advances.

V15-03 should keep the existing run feel and reward economy intact, but make the wave concept explicit:

1. Keep the first implementation's wave cadence equivalent to the current distance cadence so V15-03 is a structural change, not a balance pass.
2. Convert each 900-distance segment into a deterministic wave window.
3. Map hazards that would have appeared in that distance range into phase-owned scheduled groups.
4. Keep existing hazard types and their existing kill/breach handling.
5. Keep the existing Survival wave counter semantics so downstream rewards and unlocks still observe the same wave numbers unless a later balance task changes them.

## Wave model

Each Survival wave has four phases in this order.

| Phase | Purpose | Player-facing behavior | Scheduling rule |
|---|---|---|---|
| `telegraph` | Warn the player what kind of pressure is coming. | HUD shows the next wave number, phase label, progress bar, and concise warning text. | No surprise burst. Only prepare phase metadata and optionally show non-damaging markers through existing HUD/canvas affordances. |
| `spawn` | Introduce the wave's main threats. | Existing hazards enter in a readable sequence. | Release scheduled hazards from the wave window using seeded offsets. |
| `peak` | Create the strongest pressure. | Breach danger becomes most likely; player must defend Mine Core and survive. | Densest scheduled hazard grouping, biased toward existing non-crystal hazards that can threaten Core. |
| `lull` | Let the player recover and read the next wave. | Fewer/no scheduled hazards; HUD counts down to next telegraph. | Finish remaining delayed hazards, then transition to the next wave. |

The first implementation should define phase boundaries as proportions of the existing wave window rather than new seconds-based balance values. Recommended initial boundaries for each wave's current 900-distance window:

- `telegraph`: first 15% of the wave window.
- `spawn`: next 35%.
- `peak`: next 30%.
- `lull`: final 20%.

These proportions preserve the existing distance-based pacing while giving the code explicit phase names. If implementation discovers that exact ratios need adjustment, the implementation PR must update this section and explain why; it must not silently tune economy or rewards.

## Determinism contract

The wave director must be deterministic from the Survival run seed and existing deterministic run inputs.

Required rules:

- Initialize the director only when `currentMode === 'survival'`.
- Use a local seeded RNG stream derived from `currentSeed` and a stable wave-director salt, not `Math.random()`.
- Store all wave state on run state or run-scoped globals that are reset by `startGame()`.
- Do not read wall-clock time for wave decisions; phase progress is derived from simulation distance/ticks.
- Given the same seed and same player input stream, wave number, phase, scheduled hazards, breaches, rewards, and run end must match.
- Daily Mine and Duel must not instantiate or consult the Survival wave director.

## Required state/data

The implementation should keep the state minimal and run-scoped.

Required director fields:

- `active`: true only for Survival while a run is playing.
- `seed`: derived from `currentSeed` for reproducibility.
- `waveIndex`: 1-based current wave number.
- `phase`: one of `telegraph`, `spawn`, `peak`, `lull`.
- `waveStartDistance` and `waveEndDistance`: boundaries for the current wave window.
- `phaseStartDistance` and `phaseEndDistance`: boundaries for the current phase.
- `phaseProgress`: 0..1 normalized progress within the current phase.
- `waveProgress`: 0..1 normalized progress within the current wave.
- `nextWaveIndex`: `waveIndex + 1` while not in end-of-run handling.
- `telegraphText`: short HUD-safe label generated from scheduled pressure.
- `scheduledHazards`: references or IDs for existing hazards assigned to this wave/phase.
- `releasedHazardCursor`: index/cursor for hazards already made active or released.

Optional implementation helpers:

- `phasePlan`: compact per-wave plan generated from seed and existing hazards.
- `pressureRating`: low/medium/high label for HUD text only.
- `breachPressure`: derived label indicating whether the wave is likely to pressure the Core.

Do not persist these fields to save data. They are run state and can be rebuilt on run start.

## Hazard and enemy scheduling

V15-03 must not add new enemy types. It may only schedule existing hazard types already present in Survival, such as bats, rocks, spikes, saws, and crystals.

Scheduling should work by assigning existing generated hazards to explicit wave phases rather than inventing a new reward or enemy pipeline:

1. Generate the existing Survival hazard list deterministically.
2. For each wave window, collect hazards whose original `x` positions fall inside that window.
3. Assign non-crystal hazards primarily to `spawn` and `peak`; keep crystals as existing resource targets, not Core-pressure enemies.
4. Preserve existing hazard object shape and downstream kill/breach/reward code.
5. If a hazard is shifted within its wave window for readability, use only deterministic offsets from the director RNG and keep it inside the same wave window.
6. Do not alter kill rewards, crystal rewards, crate rewards, or run-end reward formulas.

This keeps V15-03 as a scheduling/state-machine change. Any new enemy behavior, elite modifier, boss pattern, or reward multiplier belongs to later V15 tasks.

## Mine Core breach pressure

The director should make the Mine Core feel pressured without changing Core HP values, rewards, or economy.

Allowed pressure tools:

- Concentrate existing non-crystal hazards during `peak` so breach risk is readable.
- Prefer phase telegraphs that tell the player when Core pressure is rising.
- Keep breach line and Core damage rules from V15-01 unchanged.
- Let lull phases create relief after peak pressure.

Forbidden pressure tools:

- Increasing Core damage values.
- Adding new breach-only enemies.
- Adding new rewards for defending the Core.
- Changing currency output, wave milestone rewards, or run-end reward formulas.

## HUD requirements

V15-02 added a Survival HUD with Core HP, breach state, current wave, and distance-derived next-wave progress. V15-03 implementation should feed that HUD from the explicit director instead of distance modulo math.

Required HUD display:

- Current wave number: `Wave N`.
- Current phase label: `Telegraph`, `Spawn`, `Peak`, or `Lull`.
- Phase/progress bar: fill from director `phaseProgress` or `waveProgress`, with the label making clear which progress is shown.
- Next-wave telegraph: during `lull`, show the next wave number and its upcoming pressure label; during `telegraph`, show the current incoming pressure.
- Breach warning remains driven by actual nearby hazards and Core threat state, not by a fake timer.

Recommended text behavior:

- `telegraph`: `Wave N incoming · <pressure label>`.
- `spawn`: `Wave N · enemies entering`.
- `peak`: `Wave N peak · Core pressure` when scheduled non-crystal density is high.
- `lull`: `Lull · next Wave N+1 <percent>%`.

The HUD implementation must remain readable on mobile and must not expose debug-only internal IDs.

## Transition plan for V15-03 implementation

The later implementation PR should be small and staged:

1. Add a Survival-only wave director module/helper or run-scoped functions.
2. Initialize/reset director state in Survival `startGame()` only.
3. Preserve current `survivalWave` value by deriving it from the director's `waveIndex`.
4. Replace HUD next-wave progress with director phase/wave progress.
5. Assign existing hazards to wave phases without changing their type, reward, or damage semantics.
6. Run the full verification gate and compare smoke/sim results for deterministic stability.

If any step requires economy tuning, new reward data, save migration, or non-Survival behavior, stop and split that work into a later explicitly scoped task.

## Implementation acceptance criteria

`V15-03-impl — Wave director implementation` is accepted only when all of the following are true:

- Survival uses explicit wave phases in the order `telegraph` → `spawn` → `peak` → `lull`.
- Wave number and phase are deterministic from seed and simulation progress.
- The implementation uses no `Math.random()` for wave decisions.
- Current distance-derived wave behavior is transitioned to explicit wave windows without a silent balance/economy change.
- Existing hazard/enemy types are scheduled; no new enemy types are introduced.
- No new rewards, reward multipliers, currencies, difficulty tiers, save keys, or economy values are added.
- Mine Core pressure comes from timing/readability of existing hazards, not Core HP/damage/reward changes.
- V15-02 HUD displays wave number, phase/progress, next-wave telegraph, Core HP, and breach warning from real run state.
- Daily Mine and Duel remain fair standard-build modes and do not use the Survival wave director.
- Campaign behavior remains unchanged.
- `npm run check`, `npm run balance:check`, `npm run smoke`, `npm run smoke:browser`, `npm run simulate`, `npm run build`, `git diff --check`, and `git status --short` are run and reported.

## Risks and rollback plan

Risks:

- The structural transition could accidentally change hazard density or reward timing.
- Phase scheduling could introduce nondeterminism if it uses wall-clock time or global random calls.
- HUD labels could imply fake pressure if not tied to actual scheduled hazards.
- Daily/Duel fairness could be affected if shared hazard generation is changed globally.

Rollback plan:

1. Keep V15-03 implementation Survival-only and isolated behind `currentMode === 'survival'`.
2. Preserve the existing hazard generation path so rollback can return Survival to distance-derived waves by disabling director scheduling.
3. Do not add save keys; rollback must not require migration.
4. Do not change economy values; rollback must not require balance compensation.
5. If `simulate` or browser smoke fails, revert the implementation patch and keep this design doc as the next implementation guide.
